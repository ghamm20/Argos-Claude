## Phase Z3 — Stop streaming

**Files:** `components/ChatPane.tsx`

**Demo-critical feature:** if a chat goes off the rails (LLM hallucinating, generating endlessly, model loaded wrong), the operator needs a way to halt without page refresh.

**Implementation:**
- `useRef<AbortController | null>(null)` per send invocation
- `controller.signal` threaded through `fetch("/api/chat", { signal: ... })`
- Send button swaps to a red Stop button while `isStreaming`
- Stop button → `controller.abort()` → fetch throws AbortError → caught and treated as a clean stop, not an error
- Transcript shows `_[stopped by operator]_` appended to whatever content streamed before abort
- `isStreaming` cleared, HUD metrics preserved for the partial response

**Server-side:** the chat route already had `AbortController` wiring for the upstream Ollama fetch. The client abort propagates via `reader.cancel(reason)` in the stream's `cancel` callback.

**Verification:** eyes-on (no automated browser test). Server-side abort wiring confirmed in `app/api/chat/route.ts` lines 242–246.

**Commit:** `49c9487` (combined Z2+Z3)

## Phase Z4 — `npm run warm` model preload

**Files:** `scripts/warm-ollama.mjs` (new), `package.json`, `docs/05-OPERATIONS.md`

**Problem solved:** Phase K measured 9.5s real launcher cold-start, of which 6.8s was llama3.1:8b model load. That happens on the FIRST chat of a session. Operator demo risk: "I'm sending a hello message, why is it 8 seconds…"

**Solution:** standalone CLI that pre-loads both models via:
- `POST /api/generate` with `{ model, prompt: "hi", stream: false, options: { num_predict: 1 } }` → returns when model is fully in VRAM
- `POST /api/embeddings` with `{ model, prompt: "warm" }` → returns when embed model is loaded

After running, first user-visible chat hits at sub-second TTFB.

**Behavior:**
- Reads `OLLAMA_HOST` env (matches launcher + `lib/ollama-config.ts`)
- Tolerant model-name lookup (`llama3.1:8b...` vs `name:latest` matching)
- 30s daemon-availability wait at startup
- 120s timeout on chat warm (covers slowest cold-load on weak hardware)
- Exits non-zero with actionable `ollama pull X` hint if a model is missing

**Verification:** ran against live host daemon:
```
[1/4] waiting for ollama daemon ... ready (34ms)
[2/4] listing installed models ... 4 found
[3/4] warming chat model (llama3.1:8b-instruct-q4_K_M) ... ok (7492ms, load_duration=7443ms)
[4/4] warming embed model (nomic-embed-text:latest) ... ok (1852ms, 768 dims)
```

Total: 9.3s wall time. Models then warm.

**Operator flow:**
```
Terminal 1: launcher.bat   (10s to ready)
Terminal 2: npm run warm   (10s to warm)
→ First chat: sub-second TTFB
```

## Phase Z5 — Keyboard shortcuts

**Files:** `components/ChatPane.tsx`

**Shortcuts added:**
- `Esc` (no modifier) → stop streaming if `isStreaming === true`
- `Cmd/Ctrl+K` → clear chat (same `window.confirm` guard as the button)
- `Cmd/Ctrl+E` → export chat as markdown

**Guard:** ignores keystrokes when the user is typing in:
- Any other `INPUT` or `SELECT`
- Any other `TEXTAREA` (the chat textarea itself is exempt to avoid breaking its own Cmd+Enter for send)
- Any `contentEditable` element

Prevents the operator from accidentally losing draft text or triggering the confirm dialog.

**Implementation:** window-level `keydown` listener inside `useEffect`, cleanly removed on unmount.

**Button titles updated** to surface the shortcuts on hover: "Export chat as markdown (Cmd/Ctrl+E)" and "Clear chat (Cmd/Ctrl+K)".

**Commit:** `28b3901`

## Phase Z9 — Server-side chat history persistence

**Files (new):**
- `lib/sessions.ts` — atomic write-rename + fsync, 5 MB per-session cap, 200-session list cap, strict shape validation
- `app/api/chat/sessions/route.ts` — `GET` list, `POST` upsert (create-or-update)
- `app/api/chat/sessions/[id]/route.ts` — `GET` full session, `DELETE` (idempotent)
- `scripts/smoke-sessions.mjs` — 26-case smoke covering create/read/list/update/delete/idempotency/validation

**Files modified:**
- `lib/store.ts` — added `currentSessionId`, `loadSession()`. `clearChat()` now drops session linkage (next send creates fresh session).
- `components/ChatPane.tsx` — fire-and-forget auto-save after each assistant turn completes. Save failure does not block the chat surface.

**Storage layout:**
```
ARGOS_ROOT/
  state/
    sessions/
      <16-char-hex>.json    ← one file per session
      <16-char-hex>.json.<pid>.tmp ← atomic-write transient, cleaned up
```

**Wire-shape persisted per session:**
```ts
{ version, id, title, personaId, model,
  messages: [{ id, role, content, timestamp,
               personaId?, retrievalHits?, retrievalError?, errored? }],
  createdAt, updatedAt }
```

Transient flags (`isStreaming`, HUD metrics) deliberately NOT persisted.

**Title derivation:** first user message in the session, capped at 80 chars, single-line. "New session" fallback if no user messages yet.

**Doctrine compliance:**
- **Rule 1 (no host writes):** sessions live under `ARGOS_ROOT/state/`, which goes with the USB.
- **Rule 2 (no network deps):** zero new deps.
- **Rule 3 (relative paths):** all paths derive from `argosRoot()`.
- **Rule 5 (no remote fetch):** all storage is local fs.

**Scope expansion note:** The original `methodology/decisions.md` listed chat-history persistence as v2-deferred. Z9 ships basic transcript persistence in v1, with the new decisions.md entry explaining the rationale (Rule 1 compliant, Memory page semantic-memory distinction preserved, contained 3-file change with 26-case smoke).

**Verification:** 26/26 PASS on the live smoke. Verified create/read/list/update/delete/idempotency and 400/404 validation. `createdAt` preserved across updates; `updatedAt` always advances.

**Commit:** `cfb9da0`

## Phase Z10 — History panel UI

**Files:**
- `components/SessionList.tsx` (new) — dropdown panel triggered by History icon
- `components/ChatPane.tsx` — History icon, panel state, always visible (operator may want to load a past session into a fresh view)

**Behavior:**
- Click History icon → dropdown opens at top-right of chat
- Lists sessions newest-first with persona-colored bullet, title, persona name, message count, relative timestamp
- Click a row → `loadSession()` → in-memory chat replaced with the stored session's messages, personaId, model
- Hover row → trash icon reveals → confirm + delete
- "current" pill on the active session
- Auto-refresh on open + after delete

**Memory page deliberately UNCHANGED.** The existing Memory page makes a strong doctrine point: a `Memory` page that secretly dumps the message buffer to disk would be a fake feature. Z9's chat history persistence is a different concept (basic transcript storage, time-ordered, no semantic recall). The Memory page remains the v2 stub for semantic memory + user-modeling.

**Commit:** `cfb9da0`
